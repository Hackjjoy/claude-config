AoS Internal Platform

An enterprise tool
repository, with rails.

A monorepo of internal web apps deployed to Cloudflare. Developers ship the app; the backend team owns deployment, KV configuration, and the harness guardrails that keep every tool predictable. design-review-tool is the reference example.

Designed for 5–10 tools & ~30 team members

Developer

Owns the app: UI, business logic, local testing. Brings or creates a web app, then iterates inside the repo.

Backend Team

Owns the platform: Cloudflare deployment, KV namespaces, auth, asset routing, and the skills that enforce conventions.

Access model

Access is per-tool and per-person — a team member can hold Deploy rights on one tool and only View rights on another. Accounts use email + password; no SSO is required.

Deploy rights

Tool Owner / Backend

Push new code or assets and trigger a deploy. Includes all Content and View capabilities. Typically held by the Tool Owner (Developer) for app changes, and always by the Backend team for platform changes.

Content rights

Contributor

Edit data and content inside the tool without touching code or triggering deploys. Suited for team members who manage entries, review submissions, or update configuration through the tool's UI.

View rights

Viewer

Read-only access to the tool. Can see all data and outputs but cannot make changes or trigger any actions. Useful for stakeholders, auditors, or anyone who needs visibility without edit capability.

Auth & provisioning

Method email + password
SSO not required
Scope per-tool

Accounts are provisioned by the Backend team. A user can hold different rights on different tools — Deploy on Tool X, View on Tool Y. Backend manages the access list for the platform; Tool Owners manage their own tool's contributor list within those bounds.

Ownership boundary

The line between Developer and Backend is what keeps the harness predictable. Anything in the left column is a Developer edit; anything in the right requires Backend review.

Developer edits

Developer — owns app behavior
  • src/worker.js route handlers & business logic
  • public/ static UI, HTML, assets
  • Local wrangler dev loop
  • App-level tests & data model
  • Manage access lists for their tool

Backend reviews / owns

Backend — owns platform
  • wrangler.toml: account_id, KV ids, bindings
  • Auth seam, secrets, environment
  • Asset path normalization for CF hosting
  • .github/workflows/ deploy gates
  • Provision platform accounts, assign initial rights

Harness guardrails

The repo isn't just a folder — it's a set of enforced conventions. CI fails fast when a tool drifts.

Layout check
Every tool folder must have src/worker.js, public/, wrangler.toml, package.json.
Wrangler validation
account_id set, KV namespace bound, assets directory points at ./public.
No secrets in repo
Secrets via wrangler secret. CI greps for stray tokens, hashes, account ids in PRs.
Auth seam present
Worker exposes a recognizable auth handler; new tools can't ship public-by-default.
Asset path encoding
Spaces in asset names URL-encoded — caught by a static check before deploy.
Skill-driven scaffolding
/create_tool and /migrate_tool are the only blessed entry points for new folders.
Reference implementation: design-review-tool/. Scenarios apply to every future tool that lands in this repo.
